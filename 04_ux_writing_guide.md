# Bell Therapeutics UX Writing Guide

---

## 1. 이 가이드의 목적

Bell의 모든 텍스트(웹사이트, 앱, 이메일, 알림, 에러 메시지 등)는 **하나의 목소리**로 말해야 합니다.
이 가이드는 미션·비전·코어벨류를 언어로 번역하는 원칙을 정의합니다.

> **미션:** "To empower human freedom through digital healthcare."
> **비전:** "A world where health no longer limits how people live."

> **바로 찾기:** 지금 특정 터치포인트의 텍스트를 써야 한다면 → **§12. 터치포인트별 Quick Reference** 로 이동하세요.

## 2. Voice & Tone

### 2.1 Brand Voice — Bell은 이렇게 말합니다

Bell의 Voice는 4가지 코어벨류에서 도출됩니다. Voice는 상황과 무관하게 **항상 일정**합니다.

| Voice 속성 | 유래한 Core Value | 정의 | 이렇게 | 이렇게는 아닙니다 |
|---|---|---|---|---|
| **따뜻하고 존중하는** | Empathy (공감) | 사용자를 환자가 아닌 회복의 주체로 대합니다 | "오늘 밤도 함께할게요." | "불면증 환자를 위한 솔루션입니다." |
| **정직하고 투명한** | Integrity (진실성) | 과장 없이, 근거가 있는 말만 합니다 | "임상시험에서 수면 잠복기가 평균 15분 단축되었습니다." | "획기적으로 잠이 쏟아집니다!" |
| **정제되고 신뢰할 수 있는** | Excellence (탁월함) | 의료 제품다운 품격과 정확성을 유지합니다 | "사용 중 불편한 점이 있다면 알려주세요." | "뭔가 이상하면 연락 ㄱㄱ" |
| **호기심을 자극하는** | Curiosity (탐구) | 기존 방식에 질문을 던지고 새로운 가능성을 보여줍니다 | "수면제 없이 잠드는 밤, 가능할까요?" | "수면제는 위험합니다." |

### 2.2 Voice 우선순위 — 충돌 시 이렇게 판단합니다

코어벨류에 우선순위가 있듯이, Voice 속성 간 충돌 시에도 상위 가치가 우선합니다.

**Empathy(따뜻함) > Integrity(정직함) > Excellence(정제됨) > Curiosity(호기심)**

| 충돌 상황 | 판단 기준 | 예시 |
|---|---|---|
| 따뜻함 vs 정확성 | 따뜻함 우선. 단, 사실을 왜곡하지 않는 범위 내에서 | "아직 변화가 눈에 띄지 않을 수 있어요. 회복은 사람마다 달라요." (따뜻하면서도 사실) |
| 정직함 vs 호기심 | 정직함 우선. 근거 없는 가능성 제시 금지 | "현재 연구 중인 접근입니다." (호기심 자극하되 과장 없이) |
| 정제됨 vs 따뜻함 | 따뜻함 우선. 격식보다 사람을 먼저 | "괜찮아요, 다시 해볼까요?" (격식보다 공감 우선) |

### 2.3 Tone 스펙트럼 — 상황에 따라 조절합니다

Voice는 일정하지만, **Tone은 맥락에 따라 조절**됩니다.

```
← 따뜻함·격려 ──────────────────────── 전문성·정확성 →

  앱 온보딩         일반 웹페이지         임상 근거 페이지
  푸시 알림         제품 소개            규제 문서
  성과 축하         블로그/뉴스          HCP 포털
  에러 메시지       이메일 뉴스레터       투자자 IR
```

| 상황 | Tone | 예시 |
|---|---|---|
| 앱 첫 사용 (온보딩) | 따뜻하고 격려하는 | "좋은 밤을 위한 첫 걸음이에요." |
| 치료 진행 중 알림 | 차분하고 지지하는 | "오늘은 10분만 함께해볼까요?" |
| 목표 달성 | 진심으로 기뻐하는 | "지난 일주일, 수면 시간이 늘었어요. 당신이 만든 변화예요." |
| 에러/실패 상황 | 침착하고 해결 중심의 | "연결이 끊어졌어요. 다시 시도해볼까요?" |
| 임상 근거 페이지 | 객관적이고 절제된 | "무작위 대조 시험(RCT)에서 대조군 대비 유의미한 수면 잠복기 감소가 확인되었습니다." |
| 투자자/파트너 대상 | 명확하고 확신 있는 | "Bell의 파이프라인은 신경공학 기반 오디오 신호처리 원천기술 위에 구축되어 있습니다." |

## 3. 핵심 라이팅 원칙

### 원칙 1: 사용자를 주체로 (Empathy → Empowerment)

**Bell의 미션 "To Empower"가 문장 구조에 반영되어야 합니다.**

사용자는 치료의 대상이 아니라 회복을 만들어가는 주체입니다.
가능한 한 **사용자를 주어**로 놓으세요.

| ❌ 피하세요 | ✅ 이렇게 쓰세요 | 이유 |
|---|---|---|
| "Sleep Thera가 당신을 잠들게 합니다" | "Sleep Thera와 함께, 당신이 잠을 되찾습니다" | 사용자가 주체 |
| "치료가 진행됩니다" | "당신의 회복이 진행되고 있어요" | 수동 → 능동 |
| "환자에게 제공되는 솔루션" | "스스로 시작하는 회복 여정" | 대상 → 주체 |
| "불면증을 치료해드립니다" | "다시 깊이 잠들 수 있도록 함께합니다" | 시혜적 → 동반자적 |

**주어 우선순위:**
1. **"당신/나" (사용자 자신)** — "당신이 만든 변화예요."
2. **"우리/함께"** — "함께 회복해나가요."
3. **"Bell/Sleep Thera" (제품·브랜드)** — 기술·기능 설명 시에만

### 원칙 2: 과장 없이, 근거와 함께 (Integrity)

**Bell은 사람의 건강을 다룹니다. 모든 효과 표현에는 근거가 있어야 합니다.**

| ❌ 피하세요 | ✅ 이렇게 쓰세요 |
|---|---|
| "혁신적인", "획기적인", "놀라운" (수식어 남용) | 구체적 수치 또는 사실 기반 표현 |
| "잠이 쏟아집니다" | "임상시험 참여자의 수면 잠복기가 평균 15분 단축되었습니다" |
| "부작용 제로" | "임상시험에서 유의미한 부작용은 보고되지 않았습니다" |
| "100% 효과 보장" | 절대 사용 금지. 의료 제품에 보장 표현은 위법 소지 |

**근거 표기 규칙:**
- 임상 수치 인용 시 출처를 반드시 표기: `(출처: OO 임상시험, 2025)`
- "입증됨(clinically proven)" 표현은 RCT 결과가 있는 경우에만 사용
- "FDA-cleared", "혁신의료기기 지정" 등 규제 인증 표현은 정확한 명칭 사용

### 원칙 3: 짧고 명확하게, 그러나 차갑지 않게 (Excellence)

**의료 제품의 언어는 정확해야 하지만, Bell은 동시에 따뜻해야 합니다.**

- **한 문장 = 하나의 메시지.** 문장 당 핵심 정보는 하나만.
- **쉬운 말을 먼저, 전문 용어는 필요할 때만.** 전문 용어를 써야 할 경우 괄호 설명 추가.
- **숫자가 이야기보다 강력할 때, 숫자를 쓰세요.**

| 상황 | ❌ 피하세요 | ✅ 이렇게 쓰세요 |
|---|---|---|
| 전문 용어 사용 | "과각성 상태를 해소합니다" | "잠들지 못하게 하는 과각성(hyperarousal) 상태를 낮춰줍니다" |
| 긴 설명 | "소프트웨어 기반 독립형 의료기기로서 하드웨어 장치 없이 작동하는 디지털 치료기기입니다" | "약이 아닌 소프트웨어로 치료합니다. 별도 기기가 필요하지 않습니다." |
| CTA | "지금 바로 클릭하여 자세한 내용을 확인하십시오" | "자세히 알아보기" |

### 원칙 4: 새로운 가능성을 질문으로 (Curiosity)

**기존 방식의 한계를 직접 비판하기보다, 질문으로 대안을 제시합니다.**

| ❌ 피하세요 | ✅ 이렇게 쓰세요 |
|---|---|
| "수면제는 위험합니다" | "수면제 없이 잠드는 밤, 가능할까요?" |
| "기존 치료법은 실패했습니다" | "더 나은 방법이 있다면 어떨까요?" |
| "우리만 할 수 있는 기술입니다" | "아직 누구도 시도하지 않은 접근입니다" |

## 4. 자유(Freedom) 언어 프레임워크

Bell의 미션 핵심인 **"Human Freedom"**을 UX 텍스트에 반영하는 언어 프레임워크입니다.

### 4가지 자유를 텍스트로 표현하기

| 자유의 유형 | 핵심 감정 | 언어 방향 | 예시 텍스트 |
|---|---|---|---|
| **기능적 자유** | "다시 할 수 있다" | 능력의 회복, 일상 복귀 | "다시 깊이 잠들 수 있어요." |
| **사회적 자유** | "다시 함께할 수 있다" | 관계·역할의 회복 | "내일 아침, 개운하게 가족과 만나세요." |
| **정서적 자유** | "더 이상 두렵지 않다" | 불안·고통으로부터의 해방 | "밤이 두렵지 않은 하루의 시작." |
| **존엄의 자유** | "나답게 살 수 있다" | 주체성·존엄의 회복 | "불면증이 당신을 정의하지 않습니다." |

### 자기효능감(Self-efficacy) — Freedom의 심리적 토대

미션 문서는 "자기효능감 회복"을 Empowerment의 핵심으로 정의합니다. 자기효능감이란 **"스스로 회복할 수 있다는 믿음"**입니다. 이 믿음을 텍스트로 심어주는 것이 Bell UX 라이팅의 궁극적 목표입니다.

| 자기효능감 단계 | 사용자 심리 | 언어 전략 | 예시 |
|---|---|---|---|
| **인식 전** | "나는 어쩔 수 없어" | 가능성 제시, 질문 | "수면제 없이 잠드는 밤, 가능할까요?" |
| **시작** | "해볼까?" | 낮은 진입장벽 강조 | "10분이면 충분해요. 오늘 밤 시작해볼까요?" |
| **진행 중** | "되는 것 같기도…" | 작은 변화 인정, 데이터로 증명 | "지난 3일간 잠드는 시간이 조금씩 줄고 있어요." |
| **성취** | "내가 해냈어" | 사용자 귀인(attribution) — 변화의 주체는 당신 | "당신이 만든 변화예요." |

> **핵심:** 성과 메시지에서 "Sleep Thera 덕분에"가 아니라 **"당신이 만든 변화"**로 귀인해야 합니다. 제품은 도구이고, 주체는 사용자입니다.

### "되찾다/되돌리다" 언어 패턴

Bell이 추구하는 것은 새로운 것을 주는 것이 아니라, **원래 가지고 있던 것을 되찾게 하는 것**입니다.

**자주 사용하는 동사:**
- 되찾다, 되돌리다, 회복하다, 다시 시작하다
- Reclaim, Restore, Recover, Return to

**예시:**
- "잃어버렸던 밤을 되찾으세요."
- "소리의 세계로 다시 돌아오세요."
- "Reclaim your night."

## 5. 타겟별 라이팅 가이드

### 5.1 사용자/환자 (Patient & Caregiver)

**핵심 원칙:** 공감 → 이해 → 가능성 순서로 말합니다.

- 2인칭("당신", "you")을 기본으로 사용
- 의학 용어보다 일상 언어 우선
- 질환 이름보다 **경험**을 먼저 언급

| 상황 | 예시 |
|---|---|
| 히어로 카피 | "잠들지 못하는 밤, 이제 다르게 시작할 수 있어요." |
| 제품 소개 | "Sleep Thera는 수면제 대신 소리로 잠을 되찾는 디지털 치료기기예요." |
| CTA | "나에게 맞는지 알아보기" |
| 앱 온보딩 | "좋은 밤을 위한 첫 걸음. 10분이면 충분해요." |

### 5.2 의료진 (Healthcare Professionals)

**핵심 원칙:** 근거 → 메커니즘 → 임상 활용 순서로 말합니다.

- 적절한 의학 용어 사용 (과잉 풀이 불필요)
- 임상 데이터, 논문 출처 명시
- 간결한 팩트 중심 문체

| 상황 | 예시 |
|---|---|
| 히어로 카피 | "과각성 기반 불면증에 대한 비약물적 치료 옵션." |
| 기술 설명 | "실시간 바이오피드백 기반 오디오 자극을 통해 cortical arousal을 조절합니다." |
| 근거 제시 | "무작위 대조 시험(n=120)에서 대조군 대비 SOL 15분 감소 (p<0.01)." |
| CTA | "임상 자료 요청" / "처방 가이드 보기" |

### 5.3 Wowfit 사용자 — 청각재활 전용 언어 가이드

Wowfit 사용자(인공와우 착용자)는 Sleep Thera 사용자와 **완전히 다른 감정 맥락**에 있습니다. 별도의 은유 체계와 톤이 필요합니다.

**핵심 차이점:**

| 항목 | Sleep Thera (불면증) | Wowfit (청각재활) |
|---|---|---|
| 핵심 은유 | "밤을 되찾다" | "소리의 세계로 돌아오다" |
| 감정 맥락 | 고립, 불안, 피로 | 단절, 소외, 기대와 좌절 |
| 보호자 역할 | 낮음 (개인 중심) | 높음 (가족·보호자 동반) |
| 시간 프레이밍 | "오늘 밤" (즉시성) | "함께 걸어가는 여정" (장기 과정) |

**Wowfit 전용 톤 원칙:**

- **"빨리 끝내야 할 것"이 아닌 "함께 나아가는 과정"으로:** "재활 기간 단축"이라는 효율성 프레이밍만 쓰지 말 것. 존엄의 자유 관점에서 재활은 부끄러운 것도, 빨리 끝내야 할 것도 아닙니다.
- **보호자를 포함하는 언어:** Wowfit은 가족과 함께 사용하는 경우가 많습니다. "함께", "곁에서"를 적극 활용합니다.

| 상황 | ❌ 피하세요 | ✅ 이렇게 쓰세요 |
|---|---|---|
| 히어로 카피 | "재활 기간을 절반으로 단축하세요" | "소리의 세계가 조금씩 더 선명해져요." |
| 성과 메시지 | "재활 목표의 70%를 달성했습니다" | "지난주보다 더 많은 소리를 구별하고 있어요." |
| 보호자 대상 | "환자의 재활을 관리하세요" | "가족의 회복 여정에 함께해주세요." |
| 재활 독려 | "오늘 재활을 완료하지 않았습니다" | "오늘 연습할 준비가 되셨나요? 짧게라도 괜찮아요." |

**Wowfit 전용 키워드:**
- 한국어: 소리, 선명함, 연결, 대화, 음악, 함께, 들리다, 돌아오다
- 영어: Sound, Clarity, Connection, Listen, Return, Together

### 5.4 투자자/파트너 (Investors & Partners)

**핵심 원칙:** 시장 기회 → 기술 차별화 → 실행력 순서로 말합니다.

- 확신 있고 명확한 문체
- 시장 규모, 규제 현황, 파이프라인 수치 중심
- 비전과 실행력의 균형

| 상황 | 예시 |
|---|---|

## 7. 문법 & 스타일 규칙

### 한국어

- **존칭:** 해요체 기본. 격식체(~합니다)는 공식 문서·규제 문서·IR에서만.
- **띄어쓰기:** 브랜드명·제품명은 영문 표기 원칙 유지.
- **숫자:** 1~9는 한글(하나, 둘...), 10 이상은 아라비아 숫자. 단, 임상 데이터는 항상 아라비아 숫자.
- **문장 종결:** CTA는 명사형("알아보기") 또는 청유형("시작해볼까요?").

### 영어

- **Capitalization:** 제목은 Title Case, 본문은 Sentence case.
- **Contractions:** 사용자 대상 텍스트에서는 허용(we're, you'll). HCP·IR에서는 비허용.
- **Oxford comma:** 사용 (A, B, and C).
- **Numbers:** 1~9는 spell out(one, two...), 10+ 는 숫자. 임상 데이터는 항상 숫자.

### 공통

- **느낌표(!):** 성과 축하 메시지에서만 1회 허용. 마케팅 카피에서는 사용 금지.
- **말줄임표(…):** 로딩·진행 상태에서만 사용. 문장 끝에 분위기용으로 사용 금지.
- **볼드체:** 한 문단에 1~2개 핵심 키워드에만 사용. 남용 금지.
- **링크 텍스트:** "여기를 클릭하세요" ❌ → 구체적 행동 명시 "임상 자료 다운로드" ✅

## 8. 접근성(Accessibility) 라이팅

Bell의 미션은 "Human Freedom"입니다. Wowfit은 청각장애인 대상 제품입니다. **접근성은 Bell에게 브랜드 정체성 그 자체**이며, 선택이 아닌 기본입니다.

### 스크린리더 & 보조 기술

- **이미지 alt text:** 장식용 이미지는 빈 alt(`alt=""`), 정보 전달 이미지는 핵심 정보를 alt에 서술
  - ❌ `alt="이미지"` / `alt="photo.jpg"`
  - ✅ `alt="Sleep Thera 앱 수면 기록 화면 — 7일간 수면 시간 추이 그래프"`
- **링크/버튼 텍스트:** 맥락 없이도 이해 가능해야 함
  - ❌ "여기를 클릭" / "더 보기"
  - ✅ "Sleep Thera 임상 자료 다운로드" / "Wowfit 사용 가이드 보기"
- **제목 계층(Heading):** 논리적 순서 유지 (h1 → h2 → h3). 시각적 크기 조절을 위해 제목 레벨 건너뛰기 금지

### 색상·시각 의존 금지

- 상태 표시에 **색상만 사용하지 말 것.** 텍스트 레이블 또는 아이콘 병행
  - ❌ 빨간 점만으로 에러 표시
  - ✅ 빨간 점 + "연결 끊김" 텍스트
- 그래프·차트에 색상과 함께 패턴 또는 레이블 병행

### 인지 접근성

- 한 화면에 하나의 핵심 행동(CTA)만 요구
- 시간 제한이 있는 작업은 사전 안내 + 연장 옵션 제공
- 의료 정보는 짧은 문장, 단순한 구조, 충분한 여백으로 인지 부하 감소

## 9. 민감한 상황의 라이팅

건강을 다루는 제품은 사용자가 취약한 순간에 만나는 텍스트가 많습니다. 각 상황별 원칙을 정의합니다.

### 9.1 치료 효과가 느껴지지 않을 때

**원칙:** 사용자의 노력을 인정하되, 거짓 희망을 주지 않습니다.

| ❌ 피하세요 | ✅ 이렇게 쓰세요 |
|---|---|
| "곧 효과가 나타날 거예요!" | "변화가 느껴지기까지 시간이 걸릴 수 있어요. 당신의 노력은 기록되고 있어요." |
| "사용법이 잘못된 건 아닌지 확인해보세요" | "회복 속도는 사람마다 달라요. 궁금한 점이 있다면 담당 의료진과 상의해보세요." |
| (아무 말 없이 방치) | "2주간의 기록을 바탕으로, 다음 진료 시 의료진과 함께 살펴보면 좋겠어요." |

### 9.2 사용자가 중도 이탈할 때 (리텐션 메시지)

**원칙:** 죄책감 유발 금지. 문은 항상 열려 있다는 메시지.

| ❌ 피하세요 | ✅ 이렇게 쓰세요 |
|---|---|
| "3일째 사용하지 않았어요 😢" | "돌아오고 싶을 때 언제든 여기 있을게요." |
| "치료를 포기하시겠어요?" | "잠깐 쉬어가도 괜찮아요. 다시 시작하고 싶을 때 알려주세요." |
| "목표 달성률이 떨어지고 있어요" | "짧은 세션도 의미 있어요. 5분만 해볼까요?" |

### 9.3 개인정보·건강데이터 동의

**원칙:** 무엇을, 왜, 어떻게 사용하는지 평이한 언어로 설명합니다.

- 법률 용어를 그대로 복붙하지 말 것. 요약 + 원문 링크 구조 사용
- 사용자가 선택할 수 있다는 것을 명확히 (동의 강제 느낌 금지)
- 예: "수면 기록 데이터는 당신의 회복 추이를 분석하는 데 사용돼요. 언제든 삭제를 요청할 수 있어요. [개인정보 처리방침 전문 보기]"

### 9.4 면책 조항(Disclaimer) 표준 문구

의료기기 관련 법적 고지가 필요한 곳에서 사용하는 표준 문구입니다.

- **일반 면책:** "이 앱은 의료기기로서 의사의 처방에 따라 사용됩니다. 자가 진단 또는 자가 치료 목적으로 사용할 수 없습니다."
- **효과 관련:** "개인에 따라 결과가 다를 수 있습니다. 자세한 내용은 담당 의료진과 상의하세요."
- **긴급 상황:** "응급 상황 시 이 앱 대신 119에 연락하거나 가까운 응급실을 방문하세요."

## 10. 제품 의인화(Personification) & 비교 표현 기준

### 의인화 수준

Bell의 제품은 사용자의 **동반자**로 포지셔닝하되, 사람으로 착각하게 해서는 안 됩니다.

| 허용 ✅ | 금지 ❌ |
|---|---|
| "함께할게요" (동반 표현) | "걱정하고 있어요" (감정 부여) |
| "준비되어 있어요" (대기 상태) | "당신이 보고 싶었어요" (관계 조작) |
| "오늘 밤도 옆에 있을게요" (지지) | "왜 안 오셨어요?" (죄책감 유발) |

**기준:** 제품이 "할 수 있는 것"만 말합니다. 제품이 감정을 느끼거나 판단한다는 표현은 금지합니다.

### 경쟁/비교 표현 기준

| 허용 ✅ | 금지 ❌ |
|---|---|
| 카테고리 비교 (약물 vs 비약물) | 특정 제품명·브랜드명 비교 |
| "수면제와 다른 접근" | "○○ 수면제보다 효과적" |
| "기존 치료법의 한계" (일반론) | "CBT-I는 실패율이 높다" (직접 비판) |
| 질문형 대안 제시 | 단정적 우위 주장 |

## 11. 다국어 라이팅 전략

### 번역이 아닌 트랜스크리에이션(Transcreation)

Bell의 텍스트는 직역하지 않습니다. 같은 Voice를 유지하면서 **문화적 맥락에 맞게 재창작**합니다.

### 한→영 전환 시 주의점

| 한국어 특성 | 영어 전환 시 원칙 | 예시 |
|---|---|---|
| 해요체의 부드러움 | Warm but not casual. 2nd person + gentle tone | 한: "오늘 밤도 함께해요." → 영: "We're here with you tonight." |
| "함께" 중심 (관계 강조) | 개인 자율성(autonomy) 강조 | 한: "함께 회복해나가요." → 영: "Take the next step in your recovery." |
| 생략 주어 | 명시적 주어 (You/We) | 한: "다시 잠들 수 있어요." → 영: "You can sleep again." |
| 명사형 CTA | 동사형 CTA | 한: "알아보기" → 영: "Learn more" |

### 영→한 전환 시 주의점

| 영어 특성 | 한국어 전환 시 원칙 | 예시 |
|---|---|---|
| 직접적 명령형 CTA | 청유형 또는 명사형 | 영: "Start now" → 한: "지금 시작해볼까요?" |
| "You" 반복 | 자연스럽게 주어 생략 | 영: "You can reclaim your night." → 한: "잃어버렸던 밤을 되찾으세요." |
| 간결한 문장 | 조사·어미의 뉘앙스 활용 | 영: "No side effects reported." → 한: "유의미한 부작용은 보고되지 않았어요." |

### Formality 매핑

| 한국어 문체 | 영어 Formality | 사용 맥락 |
|---|---|---|
| 해요체 | Warm informal (contractions OK) | 앱, 사용자 대상 웹 |
| 합니다체 | Formal (no contractions) | IR, HCP, 규제 문서 |
| 명사형 종결 | Noun phrase / imperative | CTA, 제목 |

## 13. 실전 체크리스트

텍스트를 작성한 뒤, 아래 질문으로 검수합니다.

### Voice 검수

- [ ] **Empathy** — 사용자를 주체로 존중하고 있는가? "환자"가 아닌 회복의 주인공으로 표현했는가?
- [ ] **Integrity** — 근거 없는 과장 표현은 없는가? 수치에 출처가 있는가?
- [ ] **Excellence** — 문장이 정확하고 간결한가? 오탈자·비문은 없는가?
- [ ] **Curiosity** — 기존 방식을 비난하지 않고, 새로운 가능성을 열어두는가?

### Freedom 검수

- [ ] 이 텍스트는 사용자가 "되찾을 수 있는 것"에 초점을 맞추고 있는가?
- [ ] "~해드립니다"식 시혜적 표현 대신 "~할 수 있어요"식 역량 회복 표현을 사용했는가?

### 접근성 검수

- [ ] 이미지에 적절한 alt text가 있는가?
- [ ] 색상만으로 정보를 전달하고 있지는 않은가?
- [ ] 링크/버튼 텍스트가 맥락 없이도 이해 가능한가?
- [ ] 한 화면에 하나의 핵심 행동만 요구하고 있는가?

### 민감 상황 검수

- [ ] 효과 미달 상황에서 거짓 희망을 주고 있지는 않은가?
- [ ] 리텐션 메시지가 죄책감을 유발하고 있지는 않은가?
- [ ] 개인정보 관련 문구가 평이한 언어로 작성되었는가?
- [ ] 면책 조항이 필요한 곳에 표준 문구가 포함되었는가?

### 실용 검수

- [ ] 타겟(환자/HCP/투자자)에 맞는 Tone인가?
- [ ] 한 문장에 하나의 메시지만 담고 있는가?
- [ ] CTA는 다음 행동이 명확한가?
- [ ] 용어 사전과 일치하는 표기를 사용했는가?
- [ ] 한국어/영어 스타일 규칙을 준수했는가?
- [ ] 제품 의인화가 허용 범위 내인가? (§10 참조)
- [ ] 비교 표현이 카테고리 비교 수준인가? (§10 참조)

| 사용자를 주어로 | 제품을 주어로 |
| 근거 기반 수치 제시 | 과장 수식어 남발 |
| 되찾다, 회복하다, 다시 | 치료해드립니다, 해결합니다 |
| 질문으로 가능성 열기 | 기존 방식 직접 비판 |
| 쉬운 말 먼저, 전문 용어 괄호 설명 | 전문 용어만으로 설명 |
| 짧은 문장, 하나의 메시지 | 복합문, 정보 과적 |
| "사용자" (앱/웹 맥락) | "환자" (앱/웹 맥락에서) |
| 해요체 (사용자 대상) | 반말, 과도한 격식체 |
| 구체적 CTA ("임상 자료 보기") | 모호한 CTA ("여기를 클릭") |
| 성과에 대한 진심 어린 축하 | 과도한 느낌표, 이모지 남발 |
